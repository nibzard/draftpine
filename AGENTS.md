# Draftpine Agent Instructions

Draftpine v3 is a static prototype compiler for agents. The source model is page JSON plus editable HTML themes; the generated prototype stays static HTML/CSS/JS in `prototype/`.

## Source Model

Edit these files:

- `pages/*.json`
- `themes/<theme>/theme.json`
- `themes/<theme>/shell.html`
- `themes/<theme>/styles.css`
- `themes/<theme>/blocks/*.html`
- `draftpine.config.json` for project-level settings

Do not hand-edit `prototype/` or `reports/` as source. They are generated by Draftpine.

## Happy Path

1. Normalize the user request into page intent: route path, title, audience, user goal, primary action, sections, states, interactions, and exclusions.
2. Add or edit one `pages/*.json` file per route.
3. Use existing blocks from `themes/default/theme.json` when possible.
4. Add a local block with `pnpm draftpine new block <name>` when the existing blocks cannot represent the screen clearly.
5. Edit block HTML and theme CSS directly. Keep templates static and use only `{{value}}`, `{{#items}}...{{/items}}`, and `{{^items}}...{{/items}}`.
6. Run `pnpm draftpine generate --json`.
7. Run `pnpm draftpine check --json`.
8. Run `pnpm draftpine eval --strict --json` for browser, screenshot, route, accessibility, overflow, runtime, and interaction checks.
9. Fix every error in source files and rerun the checks.

For another workspace, pass it as the first positional after the command, for example:

```bash
pnpm --dir /home/agent/Draftpine draftpine generate /tmp/draftpine-v3-demo --json
```

## One-Prompt Paste

When a user wants an agent-friendly happy path, have them paste a complete brief like this:

```text
Create a browsable public website prototype with Draftpine.

Product: <what the site is for>
Audience: <who visits it>
Visitor goal: <what they need to understand or decide>
Primary action: <the main CTA>
Routes: Home, Pricing, About, Contact, Resources, 404
Style: compact, modern, public-facing, inspired by Vercel/Cal.com/shadcn
Leave out: real auth, real backend calls, production integrations
```

Convert that brief directly into page JSON and theme-block edits. Do not stop at a plan unless the user asks for one. Generate, check, run strict eval, inspect screenshots, and fix every error before reporting completion.

## Theme Rules

- Blocks are editable HTML partials under `themes/<theme>/blocks/`.
- Block templates do not execute JavaScript and do not support raw HTML insertion.
- Theme shell HTML may include the Draftpine-provided theme init slot.
- Block HTML must not include `<script>` tags, inline event handlers, root-absolute local paths, or remote fetches.
- Theme CSS must avoid `width: 100vw`, negative margins, large fixed widths without containment, and root-absolute asset URLs.
- The default generated runtime owns theme toggle, tabs, filter, and modal helpers.

## Quality Bar

A prototype is acceptable when:

- generated output is static and GitHub Pages safe
- every public page has a clear H1 and primary action marker
- links resolve locally and from nested routes
- forms and controls have accessible labels or visible text
- mobile and desktop eval have no horizontal overflow
- declared interactions visibly change state
- findings point back to editable page or theme files
- `pnpm typecheck`, `pnpm test`, `pnpm build`, and strict eval for relevant fixtures pass before release work is considered done

## Deployment

Deploy only when the user explicitly asks. Run generate, check, and strict eval first, then publish the generated `prototype/` output using the repository's deployment workflow.
