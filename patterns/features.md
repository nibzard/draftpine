# Feature Patterns

Use feature patterns to organize what the product does without turning the wireframe into a generic card wall.

## Feature Bento

Use for 4-8 capabilities that should be scanned quickly.

Guidance:

- Give each card one concrete visual motif: status chip, code diff, form row, table preview, mini timeline, or placeholder screenshot.
- Keep titles as specific nouns or short claims.

## Alternating Feature Rows

Use for 3-6 major capabilities that each need room for copy and proof.

Structure:

- Text and visual alternate left/right.
- Visuals should be product artifacts, not decorative shapes.
- Each row can have a small drill-down link.

## Primitive / Product Module Stack

Use when a product has named building blocks, APIs, modules, or primitives.

Structure:

- Module name.
- One-line purpose.
- Proof artifact.
- Action such as `Try`, `View docs`, or `Use template`.

## Use-Case Bento

Use when a visitor needs to find themselves in the product.

Structure:

- 4-8 use cases.
- Each card names the job, user, or workflow.
- Optional filters only when the list is large enough to justify them.

## Product-Spec / Limits Table

Use for infrastructure, pricing, developer tools, and enterprise evaluation.

Structure:

- Rows for limits, regions, latency, retention, concurrency, roles, exports, or SLA.
- Columns compare plan, environment, current limit, or observed production behavior.
